# Security Group Sync / Reconciliation Engine

This project contains a dependency-aware AWS Security Group reconciliation tool.

The tool compares security groups exported from a source AWS account/VPC against live security groups in a target AWS account/VPC. It is designed for DR parity use cases where the target environment must contain the same required security group rules as the source environment.

> Current behavior: this tool runs in **source-required-only mode**.  
> Extra ingress or egress rules in the target account are allowed and are not treated as drift.

## What This Tool Does

The engine:

- Loads source security groups from a JSON export.
- Reads target security groups live from AWS.
- Matches source SGs to target SGs.
- Detects missing source-required ingress and egress rules.
- Remaps source SG references to the matching target SG IDs.
- Creates missing target SG shells when needed.
- Adds missing rules to the target.
- Syncs tags where applicable.
- Produces a Terraform-like execution plan.
- Writes a JSON report after each run.
- Writes rollback journal entries for reversible apply actions.

---

## Current Reconciliation Policy

This tool currently follows a **source-required-only** policy.

That means:

| Scenario | Result |
|---|---|
| Source SG exists in target | OK |
| Source SG missing in target | Reported as missing; can be created |
| Source rule missing in target | Reported as drift; can be added |
| Target has extra ingress rule | Allowed |
| Target has extra egress rule | Allowed |
| Target has all source-required rules | In sync |
| Target has AWS-managed/default extra outbound rule | Allowed |
| Target tag differs from source | Reported as tag drift |
| Target description differs from source | Reported as description drift |

In simple terms:

```text
The target must contain everything required by source.
The target may contain additional rules.
```

---

## Supported Matching Methods

The engine supports the following source-to-target SG matching methods:

1. **Exact name match**
   - Best and safest match.
   - Example: source `app-sg` matches target `app-sg`.

2. **CloudFormation tag match**
   - Uses CloudFormation logical ID and stack-name tags when available.

3. **Optional normalized-name fallback**
   - Enabled with `--allow-legacy`.
   - Helps when names differ slightly due to environment prefixes, suffixes, or generated naming patterns.
   - Ambiguous normalized matches are blocked in apply mode.

## Export Source / Prod Security Groups

Run this from the source/prod account.

Recommended export command:

```bash
aws ec2 describe-security-groups   --profile prod   --region us-east-1   --filters Name=vpc-id,Values=vpc-your-prod-vpc-id   --output json   > sg-export.json
```

Replace `vpc-your-prod-vpc-id` with the actual source/prod VPC ID.

> Important: `sg-export.json` may contain internal SG names, VPC IDs, CIDRs, account IDs, and descriptions. Do not commit real production exports into a public GitHub repository.

## Recommended First Run: Dry Run

Always start with dry-run.

```bash
python3 sg_reconcile_engine.py   --json-path sg-export.json   --target-profile dr   --target-region us-east-1   --target-vpc-id vpc-your-dr-vpc-id   --dry-run
```

Dry-run mode:

- Does not modify AWS.
- Shows what would be created or updated.
- Writes a JSON report under `sg_reports/`.

---

## Dry Run With Legacy Matching

Use this only when exact-name matching is not enough.

```bash
python3 sg_reconcile_engine.py   --json-path sg-export.json   --target-profile dr   --target-region us-east-1   --target-vpc-id vpc-your-dr-vpc-id   --dry-run   --allow-legacy
```

`--allow-legacy` enables normalized-name fallback matching.

Example normalization behavior:

```text
prod-app-web-sg-a1b2c3d4
dr-app-web-sg-e5f6g7h8
```

may normalize to:

```text
app-web-sg
```

Use this carefully. If multiple target SGs normalize to the same value, the tool marks the match as ambiguous and blocks apply mode.

---

## Report Only Mode

Use report-only when you want analysis without execution behavior.

```bash
python3 sg_reconcile_engine.py   --json-path sg-export.json   --target-profile dr   --target-region us-east-1   --target-vpc-id vpc-your-dr-vpc-id   --report-only
```

Report-only mode:

- Reads source export.
- Reads target SGs live.
- Builds the matching and drift report.
- Does not create SGs.
- Does not add rules.
- Does not sync tags.

---

## Apply Mode

Use apply mode only after reviewing the dry-run report.

```bash
python3 sg_reconcile_engine.py   --json-path sg-export.json   --target-profile dr   --target-region us-east-1   --target-vpc-id vpc-your-dr-vpc-id   --yes
```

Apply mode can:

- Create missing SG shells.
- Add missing source-required ingress rules.
- Add missing source-required egress rules.
- Sync tags.
- Write rollback journal entries.

Apply mode does **not** remove extra target rules under the current source-required-only policy.

---

## Important Behavior: Extra Target Rules Are Allowed

The current version intentionally ignores extra target ingress and egress rules.

Example:

Source egress:

```text
No outbound 0.0.0.0/0 rule
```

Target egress:

```text
All traffic to 0.0.0.0/0
```

Result:

```text
No drift, as long as all source-required rules exist in target.
```

This is useful for DR environments where the target may contain AWS-managed, DRS-managed, or environment-specific rules.

---

## What Counts as Drift

The tool reports drift when:

- A source-required ingress rule is missing from the target.
- A source-required egress rule is missing from the target.
- Tags differ.
- Descriptions differ.
- A source SG is missing from the target.
- A rule references another source SG that cannot be mapped to a target SG.

The tool does **not** report drift only because the target has additional rules.

## Reading the Report

Reports are written to:

```text
sg_reports/
```

Example:

```text
sg_reports/sg_reconcile_dryrun_dr_us-east-1_20260508_132808.json
```

Important report fields:

| Field | Meaning |
|---|---|
| `total` | Number of source SGs analyzed |
| `strict_matches` | SGs matched by exact name or strict logic |
| `legacy_matches` | SGs matched by normalized-name fallback |
| `missing` | Source SGs not found in target |
| `drifted` | SGs with missing source-required rules, tag drift, or description drift |
| `unresolved_dependencies` | SG references that could not be mapped |
| `in_sync` | SGs considered compliant with the current policy |

A clean result looks like this:

```json
{
  "total": 4,
  "strict_matches": 4,
  "legacy_matches": 0,
  "missing": 0,
  "drifted": 0,
  "unresolved_dependencies": 0,
  "in_sync": 4
}
```

This means:

```text
All source SGs exist in target.
All source-required rules exist in target.
No unresolved SG references exist.
```

## Common Output Examples

### Already In Sync

```text
[SKIP]
= SKIP app-sg (sg-source-id) :: {'reason': 'already in sync', 'target_group_id': 'sg-target-id'}
```

Meaning:

```text
The target SG already contains all source-required rules.
```

### Missing Rule

```json
"missing_ingress": [
  {
    "IpProtocol": "tcp",
    "FromPort": 443,
    "ToPort": 443
  }
]
```

Meaning:

```text
The target is missing a source-required ingress rule.
```

### Missing Security Group

```json
"missing": [
  {
    "source_name": "app-db-sg",
    "target_id": null
  }
]
```

Meaning:

```text
The source SG does not currently exist in the target VPC.
```

In apply mode, the script can create a target SG shell for it.

## Rollback Journals

Rollback journals are written under:

```text
rollback/
```

Rollback journals are generated when apply mode performs reversible actions.

Current rollback journaling supports tracking actions such as:

- Newly created SGs.
- Rules added during apply mode.

Rollback support is not yet a full automatic rollback engine. Treat the journal as an audit and recovery aid.

## GitHub Safety Warning

Do not commit real environment data into a public repository.

Avoid committing:

- Real `sg-export.json`
- Real AWS account IDs
- Real VPC IDs
- Real security group IDs
- Internal CIDRs
- Generated reports
- Rollback journals
- AWS credentials
- SSH keys
- `.env` files

Recommended `.gitignore`:

```gitignore
# Python
__pycache__/
*.pyc
venv/
.env

# AWS / secrets
.aws/
*.pem
*.key
*.crt
.env*
credentials
config

# Runtime outputs
sg_reports/*.json
rollback/*.json

# Real exports
sg-export-real*.json
prod-sg-export*.json
dr-sg-export*.json

# Keep folders
!sg_reports/.gitkeep
!rollback/.gitkeep

# OS/editor
.DS_Store
.vscode/
.idea/
```

If `sg-export.json` contains real production data, either sanitize it or remove it from Git tracking:

```bash
git rm --cached sg-export.json
echo "sg-export.json" >> .gitignore
git add .gitignore
git commit -m "Ignore real security group export"
```

## Recommended Team Workflow

1. Export source/prod SGs.

```bash
aws ec2 describe-security-groups   --profile prod   --region us-east-1   --filters Name=vpc-id,Values=vpc-prod-id   --output json   > sg-export.json
```

2. Run dry-run.

```bash
python3 sg_reconcile_engine.py   --json-path sg-export.json   --target-profile dr   --target-region us-east-1   --target-vpc-id vpc-dr-id   --dry-run
```

3. Review the report under `sg_reports/`.

4. Confirm:
   - `missing` is expected.
   - `drifted` only contains missing source-required rules or tag/description drift.
   - `unresolved_dependencies` is `0`.
   - No ambiguous matches exist.

5. Run apply mode only after review.

```bash
python3 sg_reconcile_engine.py   --json-path sg-export.json   --target-profile dr   --target-region us-east-1   --target-vpc-id vpc-dr-id   --yes
```

6. Rerun dry-run to confirm convergence.

```bash
python3 sg_reconcile_engine.py   --json-path sg-export.json   --target-profile dr   --target-region us-east-1   --target-vpc-id vpc-dr-id   --dry-run
```

Expected final result:

```json
{
  "missing": 0,
  "drifted": 0,
  "unresolved_dependencies": 0
}
```

## CLI Options

| Option | Purpose |
|---|---|
| `--json-path` | Path to source SG export JSON |
| `--target-profile` | AWS CLI profile for target/DR account |
| `--target-region` | AWS region for target/DR account |
| `--target-vpc-id` | Target/DR VPC ID |
| `--dry-run` | Preview changes without modifying AWS |
| `--report-only` | Generate report only; no execution |
| `--yes` | Apply changes |
| `--allow-legacy` | Enable normalized-name fallback matching |
| `--allow-unresolved-dependencies` | Allow execution even when SG references cannot be mapped |
| `--no-create-missing` | Do not create missing target SG shells |
| `--no-sync-tags` | Do not sync tags |
| `--report-path` | Custom report output path |
| `--rollback-dir` | Custom rollback journal directory |

Note:

```text
--revoke-extra-rules is intentionally ignored under the current source-required-only policy.
```

## Current Limitations

This engine is currently stable for source-required-only SG reconciliation, but it is still a stabilization-stage tool.

Known limitations:

- It does not delete target SGs.
- It does not remove extra target ingress or egress rules.
- It does not perform full automatic rollback.
- It does not perform advanced delete safety checks.
- It does not implement a full convergence loop.
- It does not deeply validate all AWS service-managed SG use cases.
- Complex cross-account or peered-VPC SG references may require manual review.

---

## Best Practices

- Always run dry-run first.
- Review JSON reports before apply mode.
- Avoid public commits of real AWS exports.
- Use exact SG names where possible.
- Use `--allow-legacy` only when necessary.
- Do not use `--allow-unresolved-dependencies` unless you understand the missing references.
- Keep test runs small before using this against a large environment.
- Re-run dry-run after apply mode to confirm final state.
